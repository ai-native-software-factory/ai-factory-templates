# 财务与对账（Finance & Reconciliation）业务知识

## 一、核心概念

iPlayABC 的财务系统分为两套：

| 系统 | 产品 | 特点 |
|------|------|------|
| 旧体系 | digejiaoyu1 / dige-server | 直播课订单、课时余额 |
| 新体系（剑津） | jianjin-finance | NestJS微服务，财务对账 |

---

## 二、旧体系：直播课财务

### 2.1 核心数据表

| 表名 | 用途 |
|------|------|
| live_orders | 直播课订单（购买记录） |
| live_org_extension | 机构课时余额（总课时/已用） |
| live_lessons_consume_details | 每节课的课时消耗明细 |

### 2.2 课时消耗模型

```
机构购买课时
    ↓
live_org_extension：总课时 + 已用课时
    ↓
学生上课（心跳保活，30秒一次）
    ↓
按实际上课分钟数扣减
    ↓
记录到 live_lessons_consume_details
    ↓
最小计费单位：10分钟/课时
```

### 2.3 订单支付流程

- 微信支付 / Ping++ 支付
- 支付回调 → 创建 live_order → 增加 live_org_extension 的总课时

## 三、新体系：剑津财务（jianjin-finance）

jianjin-finance 是一个基于 NestJS 的微服务，负责财务数据的统一对账和查询。

### 3.1 服务架构

```
jianjin-finance（端口8088）
  ├── api/
  │   ├── dashboard/      ── 数据概览
  │   ├── reconciliation/── 对账查询
  │   └── admin-management/── 管理功能
  └── libs/
        └── jianjin-finance/── 共享RPC服务
```

### 3.2 Dashboard（数据概览）

提供全局和各时间范围的财务指标：

| 指标 | 说明 |
|------|------|
| 激活码概览 | 总数/已激活/未激活/已作废 |
| 每日趋势 | 每日新增与激活趋势 |
| 状态分布 | 激活码状态分布 |
| 类型分布 | 按激活码类型分布 |
| 账户类型分布 | 按账户类型（kind）分布 |
| 业务模块分布 | 按模块分布 |
| 教材排名 | 各教材的激活量排名 |

### 3.3 Reconciliation（对账查询）

#### 查询维度（可多选聚合）

| 聚合维度 | 字段 | 说明 |
|---------|------|------|
| 教材 | textbook_id | 按教材分组 |
| 级别 | level_id | 按级别分组 |
| 激活码类型 | code_type | 按激活码类型分组 |
| 账户类型 | kind | 按账户类型分组 |
| 天数 | days | 按有效期天数分组 |
| 状态 | status | 按激活码状态分组 |

#### 查询条件

| 字段 | 类型 | 说明 |
|------|------|------|
| createTimeStart/End | 日期 | 创建时间范围 |
| activeTimeStart/End | 日期 | 激活时间范围 |
| textbookIds | number[] | 教材ID列表 |
| levelIds | number[] | 级别ID列表 |
| codeTypes | number[] | 激活码类型 |
| kind | string | 账户类型 |
| daysList | number[] | 天数列表 |
| statuses | string[] | 状态列表 |
| userId | number | 用户ID |
| userPhone | string | 用户手机号 |
| bookActiveCode | string | 激活码 |
| creatorId | number | 创建者ID |

#### 对账查询类型

1. **Summary（汇总）**：返回总指标（总数/求和）
2. **Aggregate（聚合）**：按维度分组，支持分页，最多返回500条
3. **Detail（明细）**：最细粒度查询，支持分页，最多返回1000条

## 四、激活码生命周期（财务视角）

```
生成（批量生成激活码）
    ↓
未激活（user_id=null，未产生费用）
    ↓
已激活（user_id=学生ID，active_date=激活日，end_date=到期日）
    ↓
已过期（end_date < 当前日期）
    OR
已作废（del=1，管理员操作）
```

### 激活码与财务的关系

- **未激活**：不产生收入确认
- **已激活**：收入确认（按天数分摊还是一次性确认，取决于财务政策）
- **已过期/已作废**：不影响已确认收入

## 六、关键发现

### 6.1 两套财务系统并存

旧体系（digejiaoyu1）的财务逻辑直接写在业务代码里（live_orders 等表），新体系（jianjin-finance）通过微服务调用获取数据。

这意味着**同一个客户可能在两套体系里都有数据**，需要数据对齐。

### 6.2 激活码是财务核心

无论是旧体系还是新体系，**激活码（book_code）是财务计数的基准**：
- 教材级别 × 激活码数量 × 单价 = 收入
- 激活码状态决定是否计入收入
