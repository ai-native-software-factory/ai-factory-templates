## 一、核心商业模式：B2B2C

iPlayABC 不是直接面向家长的产品，而是 **B2B2C 平台**：

```
┌─────────────────────────────────────────────────────────┐
│                  iPlayABC 平台                          │
│         （内容提供方 / 技术提供方）                      │
└─────────────────────────────────────────────────────────┘
         │ 卖产品/授权                 │ 提供技术支持
         ▼                              ▼
┌─────────────────────┐      ┌─────────────────────────────┐
│  机构（客户）        │      │   机构（客户）              │
│  · 雪梨教育          │      │   · 剑津教育               │
│  · 阳光全阅读        │      │   · 廖氏英语               │
│  · Bricube          │      │   · 凯狮                   │
│                     │      │                            │
│  付款买产品/课时     │      │   独立部署，定制开发        │
└─────────────────────┘      └─────────────────────────────┘
         │                            │
         ▼                            ▼
┌─────────────────────────────────────────────────────────┐
│              学生 / 家长（终端用户）                    │
└─────────────────────────────────────────────────────────┘
```

**两种 B2B 模式：**

| 模式 | 机构类型 | 平台提供 | 机构负责 |
|------|---------|---------|---------|
| **SaaS 订阅** | 标准客户 | 内容 + 技术 | 招生、运营 |
| **定制开发** | 大客户 | 内容 + 技术 + 定制开发 | 品牌、特殊需求 |

## 二、核心商业道具：激活码

激活码是整个商业模式的核心道具，串联了三个角色：

```
平台（iPlayABC）
  │ 生成激活码（batchGenerate）
  │ 卖给机构
  ▼
机构
  │ 批量生成码（batchGenerate）
  │ 嵌入图书 / 交给分销商 / 批量开通给学生
  ▼
学生/家长
  │ 扫码激活（bookActivation）
  │ 获得课程访问权
  ▼
平台
  校验码有效性 → 开通课程
```

### 激活码的规格

| 属性 | 说明 |
|------|------|
| 格式 | `{org_prefix}\|\|{16位UUID}`（如 `dige\|\|A1B2C3D4E5F6G7H8`） |
| 有效期 | 14天试用 / 365天包年 / 自定义天数 |
| 数量限制 | 一次最多 300 个 |
| 特殊规则 | `buy_540` 标签：买365天给487天（机构折扣） |

### 激活码的价值链

```
机构花钱买码（或免费获得试用码）
    ↓
机构把码放进图书 / 交给分销商 / 直接发给家长
    ↓
家长扫码 → 开通课程
    ↓
平台记录：哪个机构 × 哪个学生 × 哪个产品 × 哪天开通 × 到期日
```

**平台如何知道钱从哪来？**
- 激活码上绑定了 `creator_id`（生成者 = 机构管理员）
- 通过 `creator_id` 可以追溯是哪家机构购买的
- ⚠️ 但代码里没有看到激活码和付款的绑定关系——可能是人工对账

## 三、直播课：另一种商业模式

直播课不走激活码，走**课时包**：

```
机构购买课时包
    │
    ├── live_orders（订单记录）
    │
    └── live_org_extension（机构课时余额）
          │
          ├── 总课时
          └── 已用课时
                │
                └── live_lessons_consume_details（每节课明细）
```

### 课时消耗规则

| 规则 | 说明 |
|------|------|
| 计费单位 | 10 分钟 = 1 课时 |
| 消耗触发 | 学生进入教室，开始计时 |
| 保活机制 | 30 秒一次心跳，超时判定离开 |
| 退款 | 未消耗的课时可以退吗？**代码中未找到** |

## 四、分销系统：独立的电商业务

haoqihao 的三级分销是**独立业务**，和 iPlayABC 教育平台**不是同一套账**：

```
分销商A（level_1）
  │ 推荐
  ▼
分销商B（level_2）
  │ 推荐
  ▼
买家X 购买商品
  │ 付钱给平台
  ▼
分销商A ← 一级佣金
分销商B ← 二级佣金
分销商A ← 三级佣金（A是B的上级）
```

### 分销佣金的结算

| 状态 | 含义 |
|------|------|
| 待结算（status=0） | 佣金入账，但可退回（订单退款时） |
| 已结算（status=1） | 佣金到分销商账户 |
| 已退回（status=-1） | 订单退款，佣金退回 |
| 已取消（status=-2） | 异常取消 |

### 佣金提现方式

| 方式 | 说明 |
|------|------|
| 余额（money） | 留在平台账户，可用于下次购物 |
| 积分（score） | 积分奖励 |
| 现金（cash） | 手动打款 |
| 企业付款到零钱 | 微信自动打款 |
| 企业付款到银行卡 | 银行卡自动打款 |

## 五、打卡营：周期+内容产品

打卡营是一个**独立产品**，把课程和活动结合：

```
打卡营（tiger_clockincamp）
  ├── camp_id
  ├── syllabus_list（关联的 syllabus 列表）
  ├── start_time / end_time
  └── status_：1=未开始，2=进行中，3=已结束
```

**业务流程：**
```
机构创建打卡营
    ↓
选入 syllabus（课程内容）
    ↓
学生报名参加
    ↓
每日打卡（签到）
    ↓
完成所有课程 / 营期结束 → 判定完成
```
